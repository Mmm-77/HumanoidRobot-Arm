# 基于 AprilTag 视觉定位的四自由度机械臂上位机项目结构说明书

## 1. 文档目的

本文档依据《上位机项目规划书》整理上位机的软件目录结构、模块边界、文件职责和模块间数据关系，作为后续代码创建、任务分工、接口设计和项目验收的结构基准。

上位机采用 Linux、ROS 2 和 Python 技术栈。系统划分为以下四个相互独立的 ROS 2 软件包，每个软件包对应一个核心业务模块：

1. **视觉定位模块**：合并相机驱动与 AprilTag 解算，负责从相机图像生成有效的相机位姿。
2. **机器人运动学模块**：负责四自由度任务目标、正逆运动学、逆解筛选、关节速度和目标整形。
3. **上下位机通讯模块**：负责 USB 串口、协议编解码、目标发送、反馈接收和通信监测。
4. **总运行逻辑模块**：负责相对跟随映射、基准管理、安全状态机、数据时效检查和系统编排。

测试、配置、文档和脚本是工程支撑内容，不作为额外的软件包或业务模块。

## 2. 总体设计约定

- AprilTag 固定在环境中，相机由操作者手持移动；视觉模块输出相机相对于 `tag` 坐标系的位姿。
- 系统至少使用 `tag`、`camera`、`base` 和 `tool` 四个坐标系。
- 程序内部统一使用国际单位制：位置使用米，角度使用弧度，角速度使用弧度每秒，时间使用秒。
- ROS 2 节点之间使用带时间戳的标准消息传递数据，避免视觉处理阻塞控制和通信任务。
- 四个软件包不得直接导入其他软件包的内部 Python 实现，只能通过 ROS 2 主题、服务和标准消息交换数据。
- 每个软件包独立声明依赖、安装参数、启动入口和单元测试，支持分别构建与调试。
- 总运行逻辑模块是唯一有权切换初始化、就绪、跟随、保持、安全和故障状态的模块。
- 机器人几何参数或新版通信协议未确认时，系统必须禁止实机跟随。

## 4. 四个独立软件包说明

### 4.1 视觉定位软件包

`humanoid_arm_vision` 将相机驱动和 AprilTag 解算放在同一软件包及同一视觉节点中，减少图像跨进程复制带来的延迟。软件包内部仍保持采集、检测、位姿解算、质量判断和滤波的职责分离。

主要输入：

- 相机图像；
- 相机内参和畸变参数；
- AprilTag 家族、编号和实际边长；
- 重投影误差、检测质量和位姿跳变阈值。

主要输出：

- 相机相对于 `tag` 的带时间戳位姿；
- 视觉数据有效标志；
- 重投影误差、检测质量和无效原因；
- 用于调试的原始图像或检测叠加图像。

软件包边界：视觉包不建立跟随基准，不生成机械臂目标，也不调用串口。

### 4.2 机器人运动学软件包

`humanoid_arm_kinematics` 只处理机械臂几何模型和关节目标。总运行逻辑给出四维任务目标后，该软件包完成逆解、筛选、回代验证、关节速度计算和整形。

主要输入：

- 四维任务目标 `[x, y, z, theta]`；
- 当前四关节角；
- 机械臂模型、关节限位和速度限制。

主要输出：

- 四个目标关节角；
- 四个目标关节角速度；
- 逆解有效性、误差和奇异性信息。

软件包边界：运动学包不解释串口字节，不决定系统是否进入跟随状态，也不直接操作硬件。

### 4.3 上下位机通讯软件包

`humanoid_arm_communication` 将 ROS 2 物理量转换为协议帧，并将下位机反馈转换为上层统一状态。其他软件包不得直接访问字节缓冲区或依赖具体协议布局。

主要输入：

- 控制模式；
- 四个目标关节角和四个目标关节角速度；
- 串口及协议配置。

主要输出：

- 当前四关节角和角速度；
- 下位机运行状态、限位状态和故障状态；
- 通信在线状态、反馈时效和协议错误信息。

软件包边界：通讯包不修改运动学目标，不自行恢复跟随状态；断连或协议错误只上报总运行逻辑，由总运行逻辑决定安全动作。

### 4.4 总运行逻辑软件包

`humanoid_arm_runtime` 通过 ROS 2 接口连接另外三个软件包，是系统唯一的状态和安全决策中心。它负责在启用跟随时同时记录相机初始位姿与机械臂末端初始位姿，并将相机增量映射为任务目标。

主要输入：

- 视觉定位模块的相机位姿和有效状态；
- 机器人运动学模块的求解状态；
- 通讯模块的下位机反馈和通信状态；
- 操作者的启用、停止和故障复位请求。

主要输出：

- 四维末端任务目标；
- 跟随、保持或安全控制模式；
- 系统状态和诊断信息；
- 重新建立跟随基准的要求。

软件包边界：总运行逻辑包不实现 AprilTag 算法、逆运动学公式或串口帧编解码。

## 5. 模块间数据流

```text
humanoid_arm_vision 软件包
  相机采集 → AprilTag 检测 → 位姿解算 → 质量判断与滤波
      ↓ 相机在 tag 坐标系中的有效位姿

humanoid_arm_runtime 软件包
  基准记录 → 相对位姿 → 轴向/比例映射 → [x,y,z,theta]
      ↓ 四维任务目标

humanoid_arm_kinematics 软件包
  工作空间检查 → 逆运动学 → 逆解筛选 → 回代验证 → 目标整形
      ↓ 四关节目标角度与角速度

humanoid_arm_communication 软件包
  协议编码 → USB 串口发送 → 下位机 → 反馈解析
      ↓ 当前关节、运行状态、故障状态和通信状态

humanoid_arm_runtime 软件包
  根据反馈更新基准条件、安全状态和下一控制周期
```

### 5.1 软件包依赖方向

| 软件包 | 允许的直接依赖 | 项目内依赖规则 |
| --- | --- | --- |
| `humanoid_arm_vision` | ROS 2 图像/位姿消息、OpenCV、`cv_bridge` | 不依赖其他三个项目软件包 |
| `humanoid_arm_kinematics` | ROS 2 位姿/关节/轨迹消息、NumPy | 不依赖视觉、通讯或总运行逻辑包 |
| `humanoid_arm_communication` | ROS 2 关节/轨迹/诊断消息、pyserial | 不依赖视觉、运动学或总运行逻辑包 |
| `humanoid_arm_runtime` | ROS 2 标准消息和服务；启动时依赖另外三个包 | 只通过 launch 组合和 ROS 2 接口协调，不导入其他包的内部 Python 模块 |

三个功能执行包保持横向独立，总运行逻辑包位于编排层。任何跨包共享数据都必须转换为约定的 ROS 2 消息，不允许使用跨包全局变量、共享单例或直接访问其他包的设备对象。

## 6. ROS 2 接口

为了严格维持四个 ROS 2 软件包并避免新增接口包，四包之间优先复用标准消息：

| 数据 | 建议消息类型 | 说明 |
| --- | --- | --- |
| 原始图像 | `sensor_msgs/Image` | 由视觉模块内部或调试主题发布 |
| 相机参数 | `sensor_msgs/CameraInfo` | 保存内参、畸变和标定分辨率 |
| 相机位姿 | `geometry_msgs/PoseStamped` | `header.frame_id` 应为 `tag` |
| 四维任务目标 | `geometry_msgs/PoseStamped` | 位置为 x/y/z，姿态只表达可控 theta |
| 关节目标 | `trajectory_msgs/JointTrajectory` | 同时携带四关节位置和速度 |
| 当前关节状态 | `sensor_msgs/JointState` | 由通讯模块根据下位机反馈发布 |
| 控制模式 | `std_msgs/UInt8` | 表示安全、保持或跟随模式 |
| 启停跟随 | `std_srvs/SetBool` | 启用时由总运行逻辑检查条件并建立基准 |
| 系统诊断 | `diagnostic_msgs/DiagnosticArray` | 统一发布视觉、运动学、通信和安全状态 |

当前结构不创建第五个接口包。协议序号、故障位和无效原因优先使用诊断消息的键值字段表达；如果后续确认标准消息无法满足强类型接口要求，应单独评审是否增加接口包并同步修订本文档，而不能由某个软件包直接导入另一个软件包的内部类型。

## 7. 配置归属

| 配置文件 | 归属软件包 | 关键内容 |
| --- | --- | --- |
| `vision.yaml` | `humanoid_arm_vision` | 相机设备、内参、标签尺寸、滤波和质量阈值 |
| `kinematics.yaml` | `humanoid_arm_kinematics` | 几何参数、关节轴、工具偏置、theta 定义 |

| `communication.yaml` | `humanoid_arm_communication` | 串口、帧格式、周期、CRC、单位、重连和超时 |
| `runtime.yaml` | `humanoid_arm_runtime` | 映射矩阵、比例、偏置、死区、保持和安全恢复条件 |
